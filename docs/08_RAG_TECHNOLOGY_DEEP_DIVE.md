# 08. RAG 기술 완전 해설 — 1000페이지 PDF가 업로드될 때 실제로 일어나는 일

---

## 핵심 결론 먼저

> **"1000페이지 엔지니어링 PDF 1개 처리 = 약 $2~5 비용, 15~30분 소요"**
> 이후 질문당 응답 비용은 $0.01~0.05, 응답 속도는 5~15초.

우리 시스템은 2026년 최신 방법론 3가지를 통합했다:
- **PageIndex** (Vectify AI) — 계층 트리 기반 정밀 탐색
- **Hybrid RAG** (Vector + SQL) — 수치와 텍스트를 분리 처리
- **Query Routing** — 질문 유형별 최적 도구 자동 선택

---

## 전체 흐름 한 장 요약

```
1000페이지 PDF 업로드
        ↓
  [STEP 1] 파싱 (30분)
  pdfplumber로 페이지별 블록 추출
  텍스트 / 테이블 / 수식 / 검토결과 분류
        ↓
  [STEP 2] 3중 저장 (동시 진행)
  ┌─────────────────────────────────┐
  │ A. SQLite   — 수치 데이터       │
  │ B. Qdrant   — 벡터 임베딩       │
  │ C. JSON     — PageIndex 트리    │
  └─────────────────────────────────┘
        ↓
  [완료] 인덱싱 끝 → 질문 받을 준비
        ↓
  사용자 질문 입력
        ↓
  [STEP 3] 질문 분류 (0.5초)
  수치형 → SQL / 개념형 → Vector
        ↓
  [STEP 4] 검색 + 답변 생성 (5~15초)
  Claude API로 최종 답변 생성
```

## STEP 1: PDF 파싱 — "책을 분해하는 과정"

### 1-1. 페이지별 블록 추출

**사용 도구:** `pdfplumber` (Python 라이브러리, 무료)

1000페이지를 1페이지씩 순서대로 읽으며 각 페이지에서 블록을 추출한다.

```
페이지 1 처리:
  → 텍스트 블록 3개 발견
  → 테이블 1개 발견
  → 검토결과 패턴 2개 발견

페이지 2 처리:
  → 텍스트 블록 5개
  → 지반정수 테이블 1개 (특수 처리)
  ...
```

**소요 시간:** 페이지당 0.5~2초 → 1000페이지 = 약 15~30분

**비용:** $0 (로컬 처리, API 없음)

**토큰 소모:** 없음

---

### 1-2. 블록 유형 분류

추출된 블록을 5가지로 분류한다:

| 블록 유형 | 판별 방법 | 예시 |
|-----------|-----------|------|
| `TEXT` | 일반 텍스트 | "설계 개요 및 적용 기준..." |
| `TABLE` | pdfplumber 테이블 감지 | 일반 표 |
| `SOIL_TABLE` | 헤더에 N치, 단위중량 포함 | 지반정수 요약표 |
| `CHECK_RESULT` | `숫자 < 숫자 O.K` 패턴 | `179.83 < 270.32 O.K` |
| `SUNEX_OUTPUT` | "SUNEX", "Step No." 포함 | 전산해석 출력물 |

**왜 분류하는가?**

```
분류 안 하면:
  "179.83 < 270.32 O.K" → 그냥 텍스트로 저장
  → 나중에 "안전율 만족하나요?" 물어봐도
  → 텍스트 유사도로만 찾음 → 틀릴 수 있음

분류 하면:
  CHECK_RESULT로 저장 → {calculated: 179.83, allowable: 270.32, result: "OK"}
  → SQL로 정확하게 조회 가능
  → "179.83 < 270.32 O.K (활용률 66.5%)" 정확 반환
```

### 1-3. 테이블 구조 보존

**일반 RAG의 문제:**
```
지반정수 테이블을 텍스트로 변환하면:
"매립층 5 17.5 0 28 10000 풍화토-1 10 18.0 10 30 15000..."
→ 어떤 숫자가 어떤 지층의 어떤 값인지 불명확
```

**우리 방식:**
```python
# 테이블을 딕셔너리로 보존
{
  "지층": "풍화토-2",
  "N치": 15,
  "단위중량": 19.0,  # kN/m³
  "점착력": 10.0,    # kN/m²
  "내부마찰각": 32   # °
}
```

## STEP 2A: SQL 저장 — "수치 데이터는 반드시 DB로"

**사용 도구:** `SQLite` (로컬 DB, 무료)

파싱된 블록 중 수치 데이터를 SQLite에 구조화하여 저장한다.

### 저장되는 테이블들

```sql
-- 예시: 1000페이지 보고서 1개 처리 시 저장량

chunks            : 2,000~5,000행 (전체 청크)
```

**소요 시간:** 파싱과 동시 진행, 추가 시간 거의 없음

**비용:** $0 (로컬 처리)

**토큰 소모:** 없음

## STEP 2B: 벡터 인덱싱 — "의미 검색을 위한 준비"

### 2B-1. 청크 분할

1000페이지를 검색 가능한 조각(청크)으로 나눈다.

**청크 전략 (유형별로 다르게):**

```
TEXT 블록:
  섹션 헤더 기준으로 분리
  예: "3. 설계기준 및 지반정수" 섹션 전체가 1개 청크
  크기: 300~800 토큰

SOIL_TABLE:
  ① 테이블 전체 = 1개 청크 (전체 맥락용)
  ② 행 단위 = 지층별 각 1개 청크 (정밀 검색용)
  예: "풍화토-2 지층의 지반정수: N치=15, 단위중량=19.0kN/m³..."

CHECK_RESULT:
  단면 검토 섹션 전체 = 1개 청크
  예: "SEC-1O 단면: 발생휨모멘트 179.83 < 허용 270.32 O.K..."
```

**1000페이지 기준 총 청크 수:**
```
텍스트 청크:    약 2,000개
테이블 청크:    약 300개
행 단위 청크:   약 500개
검토결과 청크:  약 200개
─────────────────────────
합계:           약 3,000개
```

### 2B-2. 임베딩 생성

각 청크를 숫자 벡터로 변환한다. 이것이 "의미"를 숫자로 표현하는 과정이다.

**사용 API:** OpenAI `text-embedding-3-large`

```
"풍화토-2의 점착력은 10.0 kN/m²이다"
        ↓ 임베딩 API 호출
[0.023, -0.451, 0.187, ..., 0.009]  ← 3072개 숫자
```

**왜 이게 필요한가?**
```
"점착력" 검색 시:
  키워드 검색: "점착력" 글자가 있는 것만 찾음
  벡터 검색: "cohesion", "c값", "점착강도" 같은 유사 표현도 찾음
```

**배치 처리 방식:**
```
청크 3,000개를 한 번에 보내면 API 제한 초과
→ 100개씩 나눠서 30번 호출
→ 각 호출당 0.5~1초
→ 총 15~30분
```

**토큰 소모 계산:**
```
청크 1개 평균 크기:     300 토큰
총 청크:               3,000개
총 토큰:               900,000 토큰

text-embedding-3-large 가격: $0.13 / 1M 토큰
총 임베딩 비용:        약 $0.12
```

### 2B-3. Qdrant에 저장

벡터와 메타데이터를 Qdrant(벡터 데이터베이스)에 저장한다.

```
저장 형태:
{
  id: "uuid-1234",
  vector: [0.023, -0.451, ..., 0.009],  ← 3072차원

    doc_id: "abc-def",

  }
}
```

**1000페이지 기준 저장량:**
```
포인트 수:    3,000개
벡터 크기:    3,072차원 × 4바이트 = 12KB/개
총 벡터 용량: 약 36MB
```

**비용:** $0 (로컬 Qdrant, Docker)

## STEP 2C: PageIndex 트리 생성 — "목차를 AI가 만드는 과정"

### PageIndex란?

Vectify AI가 2025년 발표한 방법론. 일반 Vector RAG가 "청크 조각"을 찾는 것과 달리, **문서의 계층 구조(목차)**를 트리로 만들어서 탐색한다.

```
일반 Vector RAG:     "앵커 안전율" → 유사한 청크 5개 반환 (맥락 없음)
PageIndex:           "앵커 안전율" → 트리 탐색 → 정확한 섹션 → 정확한 답
```

### 트리 구조 생성 과정

```
1단계: 헤더 패턴으로 목차 자동 감지
  "01. 공사 개요"          → CHAPTER
  "3-2. 지반정수 산정"     → SECTION
  "3-2-1. 적용 지반정수"   → SUBSECTION
  "SEC-1O 검토 요약"       → CHECK_SECTION

2단계: 각 섹션에 LLM 요약 생성
  (claude-haiku로 저렴하게 처리)

3단계: JSON 파일로 저장
```

**생성되는 트리 예시:**
```json
{
  "title": "흙막이 가시설 설계보고서",
  "nodes": [
    {
      "title": "01. 공사 개요",
      "type": "CHAPTER",
      "pages": [1, 2, 3],
      "children": [...]
    },
    {
      "title": "03. 설계기준 및 지반정수",
      "type": "CHAPTER",
      "has_tables": true,
      "critical_values": true,
      "children": [
        {
          "title": "3-2. 지반정수 산정",
          "summary": "5개 지층 지반정수 요약표. 풍화암 kh=50,000kN/m³",
          "pages": [15, 16, 17]
        }
      ]
    }
  ]
}
```

**토큰 소모 계산 (LLM 요약 생성):**
```
요약 대상 섹션:        약 100개 (CHECK_SECTION 위주)
섹션당 입력 토큰:      500 토큰
섹션당 출력 토큰:      100 토큰
총 입력:               50,000 토큰
총 출력:               10,000 토큰

claude-haiku 가격:
  입력 $0.80/1M, 출력 $4.00/1M
총 PageIndex 비용:     약 $0.08
```

## STEP 3: 질문 분류 — "어떤 도구를 쓸지 결정"

사용자가 질문을 입력하면 **0.5초** 안에 질문 유형을 분류한다.

### 분류 로직 (LLM 없이 규칙 기반, 비용 $0)

```
"풍화토-2의 점착력은?"
  → "점착력" = 지반정수 키워드
  → NUMERICAL 분류
  → SQL 도구 선택

"앵커 안전율 기준이 뭔가요?"
  → "기준", "뭔가요" = 개념형 키워드
  → CONCEPTUAL 분류
  → Vector + PageIndex 도구 선택

"SEC-1O의 안전율이 KDS 기준 만족하는가?"
  → "만족" = 적합성 키워드
  → COMPLIANCE 분류
  → SQL + Vector 병렬 실행

"굴착깊이가 가장 깊은 단면은?"
  → "가장", "깊은" = 비교/집계 키워드
  → COMPARATIVE 분류
  → SQL GROUP BY 집계
```

## STEP 4: 검색 + 답변 생성

### Case A: 수치형 질문 → SQL 경로

**질문:** "풍화토-2의 점착력은?"

```
[1] QueryClassifier → NUMERICAL → SQL 선택

[2] NL→SQL 변환 (claude-haiku 호출)
    입력: "풍화토-2의 점착력은?" + DB 스키마 설명
    출력: SELECT cohesion FROM soil_parameters
          WHERE layer_name = '풍화토-2'

    소요: 1~2초
    토큰: 입력 300 + 출력 50 = 350 토큰
    비용: $0.0003

[3] SQLite 실행
    결과: {"cohesion": 10.0}
    소요: 0.01초
    비용: $0

[4] 답변 생성 (claude-sonnet 호출)
    입력: SQL 결과 + 질문
    출력: "풍화토-2 지층의 점착력은 10.0 kN/m²입니다.
           [출처: 흙막이보고서 p.15 / 3-2-1]"

    소요: 2~3초
    토큰: 입력 500 + 출력 150 = 650 토큰
    비용: $0.003

총 소요: 3~5초
총 비용: $0.003
정확도: 99% (DB에서 직접 조회)
```

### Case B: 개념형 질문 → Vector + PageIndex 경로

**질문:** "일시앙카와 영구앙카의 안전율 기준 차이는?"

```
[1] QueryClassifier → CONCEPTUAL → Vector + PageIndex

[2-A] 벡터 검색 (Qdrant)
    질문 임베딩 생성 (OpenAI API)
    → 토큰: 30 토큰, 비용: $0.000004
    → Qdrant에서 코사인 유사도 상위 5개 검색
    → 소요: 0.2초, 비용: $0

[2-B] PageIndex 트리 탐색 (동시 진행)
    트리 JSON 읽기 → "앵커" 관련 섹션 탐색
    → 소요: 0.1초, 비용: $0 (API 없음)

[3] 컨텍스트 조합
    Vector 검색 결과 + PageIndex 섹션 원문 합치기

[4] 답변 생성 (claude-sonnet)
    입력: 검색 결과 5개 + 질문
    출력: "일시앙카: 안전율 1.5 이상
           영구앙카: 상시 2.5, 지진시 1.5~2.0
           [출처: KDS 11 70 05 / 표 3.4 p.2]"

    토큰: 입력 2,000 + 출력 300 = 2,300 토큰
    비용: $0.01

총 소요: 5~8초
총 비용: $0.01
```

### Case C: 복합형 질문 → 모든 도구 병렬

**질문:** "굴착깊이 9m 이상 단면들의 앵커 안전율이 KDS 기준을 만족하는가?"

```
[1] QueryClassifier → COMPLIANCE → SQL + Vector 병렬

[2] 병렬 실행 (asyncio)
    ┌─────────────────────────────────────────┐
    │ Thread A (SQL)                           │
    │   굴착깊이 9m 이상 단면 조회             │
    │   → SEC-1O, SEC-1P(R), SEC-2A 반환      │
    │   → 소요 0.5초                           │
    │                                          │
    │ Thread B (Vector)                        │
    │   "KDS 앵커 안전율 기준" 검색            │
    │   → "일시앙카 SF ≥ 1.5" 조항 반환       │
    │   → 소요 0.5초                           │
    └─────────────────────────────────────────┘
    실제 소요: 0.5초 (병렬이라 합산 아님)

[3] 결과 통합
    SQL: [SEC-1O: SF=2.255, SEC-1P: SF=5.178, SEC-2A: SF=2.761]
    Vector: "일시앙카 안전율 1.5 이상 (KDS 11 70 05 4.2.3)"

[4] 답변 생성 (claude-sonnet)
    출력:
    "굴착깊이 9m 이상 3개 단면 모두 기준 만족.
     - SEC-1O: 2.255 > 1.5 ✅
     - SEC-1P(R): 5.178 > 1.5 ✅
     - SEC-2A: 2.761 > 1.5 ✅
     근거: KDS 11 70 05 조항 4.2.3
     [출처: 흙막이보고서 p.4-18, KDS 표 3.4]"

    토큰: 입력 3,000 + 출력 400 = 3,400 토큰
    비용: $0.015

총 소요: 8~12초
총 비용: $0.015
```

## 당장 추가하면 좋은 것 2가지

### 1. Re-ranking (난이도 낮음, 효과 큼)

Vector 검색 결과를 한 번 더 정렬해서 정확도를 높인다.

```python
# 현재
results = qdrant.search(query_vector, limit=10)
→ 상위 5개 바로 사용

# Re-ranking 추가 후
results = qdrant.search(query_vector, limit=20)  # 더 많이 가져옴
reranked = cohere.rerank(query, results, top_n=5)  # 다시 정렬
→ 훨씬 정확한 5개 사용

비용: 질문당 $0.002 추가
효과: 정확도 약 15% 향상
```

### 2. 청크 캐싱 (난이도 낮음, 비용 절감)

같은 문서를 여러 번 질문할 때 임베딩을 재사용한다.

```python
# Redis에 캐시
cache_key = f"embed:{doc_id}:{chunk_id}"
if redis.exists(cache_key):
    vector = redis.get(cache_key)  # 캐시 사용 ($0)
else:
    vector = openai.embed(text)     # API 호출 ($0.00004)
    redis.set(cache_key, vector)

효과: 반복 질문 시 임베딩 비용 0
```
