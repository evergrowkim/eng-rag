# 00. Doaz Engineering RAG — 프로젝트 개요

## 1. 목표

엔지니어링 문서(설계보고서, 기준서, 법령)에 특화된 RAG 시스템을 구축한다.
일반 Vector RAG의 구조적 한계를 극복하고, **수치 검증 + 기준 교차참조 + 계층 탐색**을
통합하는 2026년 방식의 하이브리드 RAG를 구현한다.

## 2. 핵심 차별점

| 일반 RAG | Doaz Engineering RAG |
|----------|----------------------|
| 텍스트 유사도로 수치 탐색 | SQL DB에서 수치 정확 조회 |
| 청크 분리로 테이블 파괴 | 레이아웃 인식으로 테이블 구조 보존 |
| 기준서 교차참조 불가 | Knowledge Graph로 조항 추적 |
| 단일 버전만 처리 | 기준 개정 이력 버전 관리 |
| 단순 텍스트 검색 | 조건부 규칙 추출 및 적합성 판정 |

## 3. 처리 대상 문서

### 3-1. 설계보고서 (Design Reports)
- 흙막이 가시설 설계 보고서
- 구조계산서
- 지반조사 보고서
- SUNEX 등 전산해석 출력물

### 3-2. 기준서 (Standards)
- **KDS** (Korea Design Standard): KDS 11 70 05, KDS 14 20 01 등
- **KCS** (Korea Construction Specification)
- **ACI** (American Concrete Institute): ACI 318-19 등
- **ASME** (American Society of Mechanical Engineers)
- **BS** (British Standards)

### 3-3. 법령 (Regulations)
- 건설기술진흥법 / 시행령 / 시행규칙
- 건축법 / 시행령
- 산업안전보건법 (가시설 관련)

## 4. 시스템 아키텍처 요약

```
[업로드] PDF/Excel/Word
    ↓
[파싱] 레이아웃 인식 → 블록 유형 분류
       텍스트 / 테이블 / 수식 / 도면 / 검토결과표
    ↓
[3중 인덱싱]
    ├─ Vector Index  → 텍스트 의미 검색
    ├─ SQL Database  → 수치 데이터 정확 조회
    └─ PageIndex     → 문서 계층 구조 탐색
    ↓
[쿼리 라우팅] 질문 유형 자동 분류
    ├─ 수치/검증 질문   → SQL
    ├─ 개념/정책 질문   → Vector + PageIndex
    └─ 복합 질문        → 병렬 멀티 도구
    ↓
[답변 생성] 엔지니어링 특화 프롬프트
    출처(페이지, 조항번호) 명시
```

## 5. 개발 로드맵

### Phase 1 — 파싱 + 저장 (Week 1)
**목표**: PDF를 업로드하면 구조화된 데이터로 저장된다

- [ ] pdfplumber 기반 레이아웃 파서 구현
- [ ] 블록 유형 분류기 (텍스트/테이블/수식/검토결과)
- [ ] SQLite 스키마 설계 및 구현
- [ ] 메타데이터 자동 추출 (프로젝트명, 기준서 참조 등)
- [ ] 파싱 결과 검증 CLI 도구

**완료 기준**: 흙막이 설계보고서 업로드 → 지반정수 테이블이 SQL로 조회됨

### Phase 2 — 벡터 인덱싱 + 기본 검색 (Week 2)
**목표**: 자연어 질문으로 관련 섹션을 찾는다

- [ ] Qdrant 연동 및 컬렉션 설계
- [ ] 청크 전략 (테이블은 행 단위, 텍스트는 섹션 단위)
- [ ] 메타데이터 필터 (문서 유형, 챕터, 페이지)
- [ ] 기본 RAG 파이프라인 (검색 + 생성)
- [ ] 검색 품질 평가 스크립트

**완료 기준**: "앵커 안전율 기준은?" → 관련 조항 + 수치 정확 반환

### Phase 3 — 쿼리 라우팅 + SQL 통합 (Week 3)
**목표**: 수치 질문은 SQL로, 텍스트 질문은 Vector로 자동 라우팅

- [ ] 쿼리 분류기 (수치형 / 개념형 / 복합형)
- [ ] SQL Tool 구현 (NL→SQL)
- [ ] 멀티 도구 병렬 실행
- [ ] 결과 통합 및 출처 명시
- [ ] FastAPI 기본 엔드포인트

**완료 기준**: "SEC-1O 근입깊이 안전율이 기준 만족하는가?" → 2.255 > 1.20 O.K 정확 반환

### Phase 4 — PageIndex + 고도화 (Week 4)
**목표**: 문서 계층 구조 기반 탐색으로 정확도 향상

- [ ] PageIndex 트리 생성기
- [ ] 계층 기반 탐색 로직
- [ ] 기준서 버전 관리
- [ ] 조항 적합성 자동 검토
- [ ] 최소 웹 UI (파일 업로드 + 채팅)

**완료 기준**: 여러 설계보고서 교차 비교 질문에 정확 답변

## 6. 성능 목표

| 지표 | 목표 |
|------|------|
| 수치 질문 정확도 | 95% 이상 |
| 텍스트 검색 정확도 | 85% 이상 |
| 응답 시간 (p95) | 10초 이내 |

## 7. 평가 방법

평가용 질문셋 (수동 작성):
```
수치형 (정답이 명확한 것):
  Q: "SEC-1O의 CIP 발생 휨모멘트는?"
  A: "179.83 kN·m"

개념형 (설명이 필요한 것):
  Q: "일시앙카와 영구앙카의 안전율 차이는?"
  A: "일시앙카 1.5, 영구앙카 상시 2.5, 지진 1.5~2.0"

복합형 (추론이 필요한 것):
  Q: "굴착깊이 9m 이상 단면들의 안전율 여유도가 가장 낮은 곳은?"
  A: SQL 집계 + 문서 맥락 통합
```
